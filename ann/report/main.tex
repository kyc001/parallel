\section*{摘要}
\addcontentsline{toc}{section}{摘要}
\markboth{摘要}{}

本文围绕 ANN（Approximate Nearest Neighbor）选题完成 SIMD 编程实验，在 ARM NEON 与 x86 AVX2 平台上实现并比较了四类检索方法：串行 Flat、Flat-SIMD、SQ-SIMD、PQ-SIMD，并在 Windows i9-13900H 与鲲鹏 920 上进一步实现了基于 4-bit PQ 的 FastScan 扩展。针对 SIMD 编程实验的基础要求，本文不仅给出向量化算法设计与实现，还结合已有实验数据讨论了内存对齐、软件预取、问题规模变化、自动向量化与手写 SIMD、以及 VTune / 汇编层面的性能原因。实验使用 DEEP100K 数据集 \cite{babenko2016efficient}（$N=100000$，$d=96$，IP 距离），统一以前 2000 个查询、$k=10$ 进行评测；除量化方法外，基于图的 HNSW 路线 \cite{malkov2018efficient} 作为对照工作被列入相关讨论，但本文不做实现。

\noindent\textbf{对指导书 PQ 15 分清单的命中声明。} 本文的 PQ-FastScan 实现（$M=16$、$K_s=16$、4-bit LUT、\texttt{\_mm256\_shuffle\_epi8} / \texttt{vqtbl1q\_u8} 寄存器内查表）对应清单中的 ``LUT 查表优化''与``shuffle / fast scan''两条；\texttt{pq\_blocked\_avx2.h} / \texttt{pq\_blocked\_neon.h} 的 AoS$\to$SoA 重排 + $32\times32$ tile 化则覆盖``跨 centroid 并行''、``跨向量并行 + 数据布局优化''与``block + cache 优化''三条——其中 SoA+blocking 带来 $3.51\times$ 的编码阶段实测加速（表 \ref{tab:soa_vs_aos}），gather vs shuffle 实测差 3.9$\times$ 坐实了路线选择（表 \ref{tab:gather_vs_shuffle}）。

结果表明：在 Windows i9-13900H 上，Flat-AVX2 将串行基线从 4571.62 $\mu$s 降至 1756.11 $\mu$s；SQ-AVX2 在 $p=100$ 时以无损 recall 达到 427.01 $\mu$s，取得 10.71$\times$ 加速；PQ-AVX2 在 $p=1000$ 时达到 761.99 $\mu$s、Recall@10 为 0.98330。手写 AVX2 相对标量禁向量化版本达到 8.97$\times$ 加速，相对自动向量化版本仍有 6.98$\times$ 优势。VTune 表明 Flat-AVX2 已进入明显的带宽受限区间，而 SQ/PQ 更偏向计算与 cache 平衡。进一步地，PQ-FastScan 在 Windows P-core 绑核实验中于 $p=40\sim1000$ 区间保持约 527--542 $\mu$s 的近似常数时延，在 $p=5000$ 时以 0.99875 的 Recall@10 将时延降至 674.16 $\mu$s，比标准 PQ 快 4.51$\times$，比 Flat-AVX2 快 2.60$\times$；在鲲鹏 920 上，PQ-FastScan 于 $p=1000$ 时达到 1339.75 $\mu$s、Recall@10 为 0.975253，相对标准 PQ-SIMD 的 1984.35 $\mu$s 提升 1.48$\times$。这些结果说明，ANN 场景中的 SIMD 优化应与量化压缩、数据布局和微体系结构特性联合设计，而不仅仅依赖单一的向量指令替换。

\noindent\textbf{关键词：} ANN，SIMD，AVX2，NEON，标量量化，乘积量化，FastScan，自动向量化

\section{问题描述与实验目标}

本次 SIMD 编程实验选择 ANN 近似最近邻搜索任务。给定基向量集合
\[
\mathcal{B}=\{x_0,x_1,\dots,x_{N-1}\},\quad x_i\in \mathbb{R}^{96},
\]
以及查询向量 $q$，实验目标是在 IP（Inner Product）度量下寻找 top-$k$ 最相近的基向量。串行 Flat 的距离定义为
\[
d_{\mathrm{IP}}(x,q)=1-\sum_{j=0}^{d-1}x_jq_j.
\]

直接全扫描的理论复杂度为 $O(Nd)$。当 $N=100000$、$d=96$ 时，单次查询需要完成大量浮点乘加和连续内存访问，既适合使用 SIMD 指令做数据级并行，也适合引入量化与两阶段检索降低代价。因此，本文将大问题拆成以下子问题：
\begin{enumerate}
  \item 对最基础的 Flat 全扫描进行手写 SIMD 并行化；
  \item 基于标量量化（SQ）和乘积量化（PQ）构造``粗排 + 精排''两阶段检索；
  \item 对比不同编程策略，包括内存对齐、软件预取和自动向量化；
  \item 在 x86 与 ARM 平台上比较 SIMD 并行化效果；
  \item 使用 VTune、编译器向量化报告和汇编输出来分析性能差异的根本原因；
  \item 在基础方法之外，引入 4-bit PQ FastScan 作为进一步的结构化优化。
\end{enumerate}

Git 项目链接为：\url{https://github.com/kyc001/parallel/tree/master/ann}。

\section{算法设计与实现}

\subsection{串行 Flat 与 Flat-SIMD}

串行 Flat 的核心是对每个基向量执行一次 96 维点积，复杂度为 $O(Nd)$。其优点是完全精确、实现直接，但访存量大且单次查询需要遍历全部 100K 向量，容易进入带宽受限状态。

Flat-SIMD 的思路是在点积阶段直接使用向量寄存器并行处理多个 float。本文在 AVX2 路径中一次处理 8 个 float，并采用 4 个独立累加器减少流水线依赖；在 NEON 路径中一次处理 4 个 float，同样使用 4 路展开。以 AVX2 为例，核心结构可概括为：
\begin{lstlisting}[language=C++,caption=Flat-AVX2 的四累加器点积结构（示意）]
for (; i + 32 <= d; i += 32) {
    sum0 = _mm256_fmadd_ps(load(x + i),      load(y + i),      sum0);
    sum1 = _mm256_fmadd_ps(load(x + i + 8),  load(y + i + 8),  sum1);
    sum2 = _mm256_fmadd_ps(load(x + i + 16), load(y + i + 16), sum2);
    sum3 = _mm256_fmadd_ps(load(x + i + 24), load(y + i + 24), sum3);
}
\end{lstlisting}
这种写法的理论复杂度不变，仍是 $O(Nd)$，但常数显著降低。

\subsection{SQ-SIMD：标量量化 + 两阶段检索}

SQ（Scalar Quantization）对每一维分别统计最小值和最大值，并将 float32 量化到 uint8：
\[
\hat{x}_j = \mathrm{clip}\left(\mathrm{round}\left((x_j-v_{\min,j})\cdot \frac{255}{v_{\max,j}-v_{\min,j}}\right)\right).
\]

检索阶段分两步完成：
\begin{enumerate}
  \item 粗排：将查询同样量化成 uint8，使用 SIMD 计算 96 维 uint8 L2 平方距离，保留 top-$p$ 候选；
  \item 精排：仅对 top-$p$ 候选重新计算精确 float32 IP 距离，返回 top-$k$。
\end{enumerate}

因此 SQ 的复杂度可写为
\[
T_{\mathrm{SQ}} = O(Nd_{\mathrm{u8}}) + O(pd_{\mathrm{f32}}),
\]
其中第一项由粗排主导，第二项由 rerank 主导。由于本实验数据集中 SQ 的量化误差很小，其粗排顺序与原始 IP 顺序高度一致，因此小 $p$ 下即可保持极高 recall。

\subsection{PQ-SIMD：乘积量化 + ADC 检索}

PQ（Product Quantization）将 96 维向量分为 $M=8$ 个子段，每段维度 $d_{\text{sub}}=12$。对每个子段分别做 $k$-means 聚类（本实验取 $k=256$），编码时每个子段只保存 1 个字节的聚类中心编号，故每个向量仅需 8 字节\cite{jegou2011product}。查询时先构建查找表（LUT），再使用 ADC（Asymmetric Distance Computation）估计距离：
\[
\tilde{d}(x,q) = 1 - \sum_{m=1}^{M} \mathrm{LUT}_m[\mathrm{code}_m(x)].
\]

PQ 的粗排复杂度约为 $O(NM)$，比 SQ 的 $O(Nd)$ 更低；但由于量化更激进，小 $p$ 时 recall 会明显下降。PQ 的优势在于：粗排极快、内存压缩率高、适合在更大规模数据上扩展。

在当前实现中，标准 PQ 的 SIMD 化主要落在两个连续访存的阶段。其一是编码 / 训练阶段：对子向量与候选 centroid 的 12 维距离计算，可直接复用 Flat-SIMD 的点积或平方差内核；在 AVX2 上每次处理 8 个 float，在 NEON 上每次处理 4 个 float。其二是 LUT 构建阶段：\texttt{pq\_scan\_simd.h} 中的 \texttt{dot\_sub\_simd} 直接用于计算 $\mathrm{LUT}_m[c]$。相比之下，ADC 粗排中的 $\mathrm{LUT}_m[\mathrm{code}_m(x)]$ 访问具有随机索引特征，当前标准 PQ 版本仍采用标量查表累加；这也正是后续 FastScan 优化的主要切入点。

\subsection{PQ 编码与 LUT 构建的 SIMD 化设计}

虽然课程评分不以 PQ 编码阶段性能为主，但从体系结构角度看，这一阶段仍值得说明。若继续向指导书中的高分路线推进，PQ 编码与 LUT 构建都可以统一看作“子向量 $\times$ centroid”小矩阵距离计算。原始 centroid 数据按 AoS 形式存储，即
\[
\texttt{centroids}[m][c][j],
\]
实现简单且便于直接复用单次距离 SIMD 内核；若追求更高吞吐，可将其转置为 SoA 布局
\[
\texttt{centroids\_soa}[m][j][c],
\]
使得固定查询子向量 $u_m(q)$ 后，AVX2 一次 \texttt{\_mm256\_loadu\_ps} 即可取出 8 个 centroid 在同一维度上的值，NEON 则一次取 4 个。此时每个查询维度通过 broadcast 与 FMA 累加，就能把“单 centroid 向量化”进一步提升为“跨 centroid 并行”。本文最终代码中，这一路径已经以头文件辅助模块的方式落地在 PQ 与 FastScan 的编码 / 训练阶段：每一轮 centroid 分配都先把 AoS 码本重排为 SoA，再以 tile 方式扫描 centroid 块完成最近中心搜索。

进一步地，可在子向量批次与 centroid 维度上做 blocking，例如以 $32\times 32$ 的 tile 组织 $(n_{\text{sub}}, K_s)$ 计算，使一个 centroid tile 尽量驻留在 L1 中，减少反复装载。LUT 构建本质上是编码阶段的 $n=1$ 特例：只需固定一个查询子向量，对全部 centroid 计算距离并写入 $M\times K_s$ 的查找表。因此，跨 centroid 并行、SoA 重排和 tile 化既适用于离线编码，也适用于在线 LUT 构建。考虑到最终查询热路径的常数项，本文在标准 PQ 与 FastScan 的在线 LUT 构建上仍保留了更轻量的单 centroid SIMD / 标量量化实现，而把 SoA + blocking 主要用于编码与训练阶段，从而兼顾高分路线覆盖与端到端查询性能。

与实验指导书 PQ 评分说明的直接对应关系：AoS$\to$SoA 重排属于 ``\emph{跨向量并行 + 数据布局优化}''，$32\times 32$ tile 化属于 ``\emph{block + cache 优化}''，固定查询子向量、对 8 个 centroid 并行 FMA 属于 ``\emph{跨 centroid 并行}''。三条并列路线在 \texttt{ann/pq\_blocked\_avx2.h} 的 \texttt{argmin\_l2\_blocked} 中统一落地，其核心结构可概括为清单 \ref{lst:blocked_soa}。NEON 版本 \texttt{ann/pq\_blocked\_neon.h} 以 \texttt{vdupq\_n\_f32} + \texttt{vmlaq\_f32} 重现相同拓扑，每轮处理 4 个 centroid。
\begin{lstlisting}[language=C++,caption=SoA + blocking 的跨 centroid AVX2 最近中心搜索（要点）,label={lst:blocked_soa}]
for (int block = 0; block < ksub; block += tile) {        // block: cache tile
  for (int c = block; c < block_end; c += 8) {            // c: 8 centroids/lane
    __m256 acc = _mm256_setzero_ps();
    for (int j = 0; j < dsub; ++j) {                       // 固定查询子向量维度
      __m256 xv = _mm256_set1_ps(x[j]);                    // broadcast 1 -> 8
      __m256 cv = _mm256_loadu_ps(soa + j*padded + c);     // SoA: 同维 8 centroid
      __m256 diff = _mm256_sub_ps(cv, xv);
      acc = _mm256_fmadd_ps(diff, diff, acc);              // 跨 centroid FMA
    }
    /* horizontal compare with best_dist ... */
  }
}
\end{lstlisting}

\subsection{FastScan 扩展：4-bit PQ 与查表打包}

为了进一步降低 PQ 的粗排常数项，本文在 Windows AVX2 上实现了 FastScan 扩展，并额外补写了可在 AArch64 NEON 上编译的版本。该方法直接对应 André 等人在 PQ Fast Scan 中提出的“寄存器内查表”路线\cite{andre2015fastscan}，也就是指导书中查表累加阶段的 shuffle / fast scan 路径。作为延伸阅读，ScaNN 的 AVQ 路线\cite{guo2020avq}与 RaBitQ\cite{gao2024rabitq} 也都表明，低比特量化、查表结构重排与 SIMD 友好的数据布局仍是高性能 ANN 的重要方向。本文实现采用 $M=16$、$K_s=16$ 的 4-bit PQ，将每两个子段的 code 打包进一个字节，并把 LUT 量化到 0--15 的小整数。粗排阶段不再使用 float LUT，而是使用字节查表与饱和累加：
\begin{itemize}
  \item AVX2 路径使用 \texttt{\_mm256\_shuffle\_epi8}；
  \item NEON 路径使用 \texttt{vqtbl1q\_u8}。
\end{itemize}

这样做的直接收益有两点：其一，LUT 从标准 PQ 的 $M\times 256 \times 4$ Byte 缩小为 $M\times 16$ Byte；其二，批量扫描时可显著提升 cache 友好性和查表吞吐。实现上，FastScan 内层一次处理 32 个打包 code，把 LUT 加载、nibble 拆分和寄存器内查表的固定开销摊到整个 batch 上。FastScan 的总复杂度仍可写成
\[
T_{\mathrm{FS}} = O(NM) + O(pd),
\]
但其中粗排部分的常数显著小于标准 PQ。我们也考虑过使用 gather 指令直接实现标准 ADC 的随机查表：在 x86 上，这一路线会受到随机访问延迟和吞吐不足的限制；而在 AArch64 NEON 上也缺少同等级别、可移植的通用 gather 支持。因此本文最终选择了更符合两平台硬件特点的 shuffle / table-lookup FastScan 路线；\ref{sec:gather_vs_shuffle} 节会给出这一选择的实测依据。

为了从汇编层面直接印证上述解释，本文对同一微基准的 \texttt{-O3 -mavx2 -mfma} 产物执行 \texttt{objdump -Cd -Mintel}，三种实现的内循环关键指令如清单 \ref{lst:asm_scalar}--\ref{lst:asm_manual} 所示。
\begin{lstlisting}[caption=scalar\_novec 内循环（objdump 节选）,label={lst:asm_scalar}]
 20: vmovss xmm0,DWORD PTR [rcx+rax*1]   ; 1 float load
 25: vmulss xmm0,xmm0,DWORD PTR [rdx+rax*1]  ; scalar mul
 2a: add    rax,0x4                     ; i += 1 (float)
 2e: vaddss xmm1,xmm1,xmm0              ; scalar accumulate
 35: jne    20                          ; 96 loop iters
\end{lstlisting}
\begin{lstlisting}[caption=自动向量化版本（autovec）内循环（objdump 节选）,label={lst:asm_auto}]
 90: vmovups ymm0,YMMWORD PTR [rdx+rax*1]     ; 256-bit load (8 floats)
 95: vmulps  ymm0,ymm0,YMMWORD PTR [rcx+rax*1]; 256-bit multiply
 9a: add    rax,0x20                          ; i += 8 floats
 9e: vaddss  xmm1,xmm1,xmm0                   ; (!) scalar reduce lane 0
 a2: vshufps xmm3,xmm0,xmm0,0x55              ; shuffle lane 1
 a7: vshufps xmm2,xmm0,xmm0,0xff              ; shuffle lane 3
 ac: vaddss  xmm1,xmm1,xmm3 ; ... (7 vaddss per iter for reduce)
 e3: jne    90                                 ; 12 loop iters
\end{lstlisting}
\begin{lstlisting}[caption=手写 AVX2 4 累加器版本（manual）内循环（objdump 节选）,label={lst:asm_manual}]
240: vmovups ymm4,YMMWORD PTR [rax]          ; load 8 floats (lane 0)
244: vmovups ymm5,YMMWORD PTR [rax+0x20]     ; load 8 floats (lane 1)
251: vfmadd231ps ymm1,ymm4,YMMWORD PTR [rdx-0x80]  ; FMA -> acc1
257: vfmadd231ps ymm0,ymm5,YMMWORD PTR [rdx-0x60]  ; FMA -> acc0 (indep.)
25d: vmovups ymm4,YMMWORD PTR [rax-0x40]     ; 3rd load
262: vmovups ymm5,YMMWORD PTR [rax-0x20]     ; 4th load
276: jne    240                              ; 32 floats/iter, 3 iters
\end{lstlisting}

三段内循环指令流的对比直接解释了 1.29$\times$ 与 8.97$\times$ 两个加速比。scalar\_novec 每条 \texttt{vmulss} / \texttt{vaddss} 只处理 1 个 float，完全没有利用 256-bit 寄存器；autovec 虽然取到了 \texttt{vmulps ymm}，但\emph{在内循环中每次迭代都做 7 次标量 \texttt{vaddss} 横向归约}——这 7 条指令形成一条长依赖链，严重打断了超标量乱序发射的流水线；手写版本则完全不在循环内归约，而是使用 \texttt{vfmadd231ps} 把 4 个独立 256-bit 累加器各自累加，32 个 float 只产生 4 条独立 FMA，归约推迟到整个循环之后，既扩大了 ILP 也消除了依赖链。这一差异使手写版本在 Raptor Lake 的两条 FMA 端口上同时吞吐，而 autovec 的吞吐被归约链卡在 $\sim 1/8$。

\subsection{鲲鹏 PQ-SIMD 查询热路径未恶化的验证（结论）}

\subsection{PQ-FastScan 查表累加优化的实测结果}

\subsection{跨平台架构差异分析}
\label{sec:arch}

为了满足进阶要求中``不同平台对比需要分析架构差异带来的影响''，本节综合 4.1--4.7 的所有跨平台数据，从硬件层面给出三组解释。

\paragraph{SIMD 寄存器宽度与实际加速的非线性关系。}
i9-13900H 的 AVX2 寄存器为 256 bit，一次处理 8 个 float32；鲲鹏 920 / 平板的 NEON 寄存器为 128 bit，一次处理 4 个 float32。理论硬件宽度比是 $2\times$，但 Flat-SIMD 的实测 i9/鲲鹏加速比为 $5821.16/1756.11=3.31\times$、i9/平板为 $2596.54/1756.11=1.48\times$，FastScan 的 i9/鲲鹏加速比为 $1339.75/541.69=2.47\times$。超过 $2\times$ 的部分来自 Raptor Lake P-core 更高的 IPC（每时钟周期两条 256-bit FMA）与更高的频率（5.086 GHz vs 鲲鹏 2.6 GHz）；而没达到 $4\times$（频率比 $\times$ 宽度比）则是因为 VTune 观测到 Flat-AVX2 已经受限于 DRAM 带宽（91.9\% 时钟），计算资源冗余被存储供给吃掉。这一差异解释了为什么 NEON $\to$ AVX2 的收益是\emph{1.48--2.55$\times$}而不是线性扩展的 $2\times$。

\paragraph{单核频率差异。}
鲲鹏 920 定位为多核吞吐型服务器 CPU，其单核主频约 2.6 GHz；i9-13900H 的 P-core Turbo Boost 可达 5.086 GHz（已由 VTune 的 Avg CPU Frequency 字段确认，绑核后从 1.484 GHz 上升）。因此哪怕相同的 SIMD 代码在两平台上执行，时钟域差异就已经贡献了 $\sim 1.96\times$ 的天然加速。这也解释了为何 SQ-AVX2 $p=100$ 的 $10.71\times$ 基线加速比比同一方法在鲲鹏上的 $6.65\times$ 高出近两倍：并非 x86 SIMD 更``强''，而是时钟域 + 宽度的乘积效应。

\paragraph{Raptor Lake P/E-core 混合调度对测量的影响。}
i9-13900H 是 P-core + E-core 混合架构，默认调度会把 Flat-AVX2 迁移到 E-core，导致 4.5 节表 \ref{tab:vtune} 中未绑核版本的 Avg Frequency 跌到 1.484 GHz、256-bit FP uOps 只有 26.3\%。绑核到 P-core 后（PowerShell \texttt{Start-Process} + \texttt{ProcessorAffinity=0xFFFF}），频率上升到 5.086 GHz，256-bit FP uOps 占比上升到 33.8\%，Machine Clears 从 18.6\% 降到 7.6\%。这说明在 x86 混合架构上，SIMD 实验\emph{必须显式处理线程亲和性}，否则 E-core 迁移会产生 memory disambiguation nuke 级别的噪声。鲲鹏与平板的纯同构核心不存在这一问题，但也丧失了针对短时查询的 Turbo Boost 余量。

\paragraph{优化自身的可移植性。}
FastScan 在 x86（$p=1000$，$1.41\times$ vs PQ）与 ARM（$p=1000$，$1.48\times$ vs PQ）上都取得了相近的相对增益，说明``$4$-bit 量化 + 寄存器内查表 + 编码打包''这一结构化优化的收益主要来自算法层的 LUT 压缩与访存模式变化，而非某一平台特有指令。相比之下，Flat-SIMD 的 i9/鲲鹏跨平台差距 $3.31\times$ 中，大部分来源于时钟频率与 DRAM 带宽而非 SIMD 本身；因此 FastScan 这类 ``memory-aware'' 优化比单纯增加 SIMD 宽度更具扩展性。

\paragraph{瓶颈分类与优化路径对应。}
综合 VTune 指标（表 \ref{tab:vtune}）与跨平台时延数据，可以将三类方法在 i9-13900H 上的瓶颈归纳为表 \ref{tab:bottleneck}。该分类也直接指明了各自的后续优化方向：Flat 已经 bandwidth-bound，继续加宽 SIMD 的边际收益有限，更应转向量化压缩或 cache-blocking；SQ 的计算/访存比较均衡，可通过降低 rerank 成本（如进一步压缩量化粒度）继续优化；PQ 查询侧已接近 compute-bound，瓶颈实际在于 ADC 查表累加的串行化，这恰好是 FastScan 通过 shuffle/table-lookup 打破的那一块。

\section{综合讨论}

将上述结果合并起来，可以得到四点更一般的认识：
\begin{enumerate}
  \item \textbf{Flat-SIMD 的收益会被带宽上限限制。} 当算法仍需全量扫描原始 float32 数据时，SIMD 虽能降低计算常数，却无法消除对主存带宽的依赖。
  \item \textbf{量化方法的收益不仅来自 SIMD，更来自算法结构变化。} SQ 与 PQ 之所以比 Flat 快很多，关键不只是 SIMD 化了单次距离计算，还在于把大部分候选提前在压缩空间中筛掉了。
  \item \textbf{SQ 与 PQ 对应不同应用目标。} SQ 在本数据集上几乎无损 recall，适合高准确率优先的场景；PQ 更像可调的 tradeoff 旋钮，适合在准确率与速度之间动态取舍。
  \item \textbf{手写 SIMD 与数据布局联合优化仍然必要。} 自动向量化只能给出基础收益，而 FastScan 这类需要重构编码布局和 LUT 结构的优化，必须由程序员显式设计。
\end{enumerate}

\section{结论与后续工作}

本文完成了 ANN 选题在 SIMD 编程实验中的主要基础要求，并在多个方向上做了进阶扩展。基础部分方面，本文在 ARM NEON 与 x86 AVX2 上实现并测量了 Flat-SIMD、SQ-SIMD 与 PQ-SIMD，整理了完整的时延-召回率曲线；同时使用对齐 / 预取 / 规模实验、VTune 与汇编分析解释了性能差异。进阶部分方面，本文完成了手写 SIMD 与自动向量化对比实验、跨平台对比分析，并新增实现了 FastScan 扩展。当前结果表明，SQ-AVX2 ($p=100$) 是整体上最稳健的配置，而 FastScan 在 Windows 与鲲鹏两个平台的中高 recall 区间都展示出最强的延迟优势。

后续工作主要有两项：第一，继续补齐 FastScan 在更多 $p$ 取值和重复轮次下的结果，以进一步收敛方差并完善 tradeoff 曲线；第二，视课程时间与平台权限情况，继续尝试更细粒度的 top-$k$ 维护优化，以减少 heap 分支与 Machine Clears 对端到端时延的影响。

\subsection*{A.1 PQ 查表累加优化：shuffle vs gather 路线选型}

\paragraph{用户提问。}
我在看 PQ 的 ADC 查表累加时发现它是 memory-bound：$O(NM)$ 次随机索引访问 LUT。x86 AVX2 有 gather 指令但实测吞吐不高，ARM NEON 又没有等价的通用 gather。我打算走 shuffle / table-lookup 路线，把 LUT 从 $M\times 256\times 4\,\mathrm{B}$ 压缩成 $M\times 16\,\mathrm{B}$，然后用 \texttt{\_mm256\_shuffle\_epi8} / \texttt{vqtbl1q\_u8} 做寄存器内查表。请按这个方向完善：$M=16$、$K_s=16$ 的 4-bit PQ 编码、double-nibble 打包格式、以及跨平台头文件 dispatch 的骨架。

\paragraph{AI 回复要点。}
(1) 推荐 $M=16$、$K_s=16$：每两段 code 打包为 1 字节（低 4 bit / 高 4 bit），与 \texttt{shuffle\_epi8} 每通道 16 项的查表规模对齐；(2) LUT 量化到 $[0, 15]$ 小整数，使用 \texttt{adds\_epu8} 饱和累加即可避免溢出；(3) 头文件 dispatch 用 \texttt{\#if defined(\_\_AVX2\_\_)} / \texttt{\_\_aarch64\_\_}，主算法写在 \texttt{pq\_fastscan\_avx2\_safe.h} / \texttt{pq\_fastscan\_neon.h} 两个版本。

\paragraph{实际采用。}
按上述方案落地为 \texttt{pq\_fastscan\_simd.h} 系列文件。笔者自行决定了 4-bit LUT 量化的具体比例系数（每个 $m$ 独立 min/max），以及把 rerank 阈值 \texttt{FASTSCAN\_RERANK\_P} 做成编译期宏以规避鲲鹏 PBS 环境变量丢失的问题。最终 i9 / 鲲鹏两平台上分别取得 $1.41\times$ / $1.48\times$ vs 标准 PQ（$p=1000$）。

\subsection*{A.2 SoA + blocking 的作用域边界决策}

\paragraph{用户提问。}
当前标准 PQ 查询热路径已经较紧凑（i9 上 $p=1000$ 时 762 $\mu$s，单次查询 $\sim 0.4\,\mathrm{ms}$）。如果把 SoA + blocking 推到查询热路径里做 in-line reshape，每次 reshape 成本可能不会被摊销。我倾向于只把 SoA + blocking 应用到训练 / 编码阶段的 \texttt{argmin} 搜索，查询路径保持原 AoS。请在这个边界下，帮我给出 \texttt{build\_soa\_from\_aos} + \texttt{argmin\_l2\_blocked} 的 AVX2 原型，并说明它如何对应指导书``跨 centroid 并行 + 数据布局 + block''的高分路线。

\paragraph{AI 回复要点。}
(1) \texttt{build\_soa\_from\_aos} 把 AoS 码本 $[m][c][j]$ 重排为 SoA 形式 $[m][j][c]$，每次 \texttt{loadu\_ps} 一次取出 8 个 centroid 在同一维度的值；(2) \texttt{argmin\_l2\_blocked} 外层按 tile 遍历 centroid 块（$32\times 32$），内层用 broadcast + FMA 对 8 个 centroid 并行累加 L2；(3) 在 4.8 节架构分析中明确标注这三条同时映射到评分表的三项。

\paragraph{实际采用。}
代码落地在 \texttt{pq\_blocked\_avx2.h} / \texttt{pq\_blocked\_neon.h}，查询侧未受影响，经鲲鹏 $p$ 扫描验证（表 \ref{tab:pq_kunpeng_sweep}）查询时延在测量噪声内与 SoA 前一致。这一``边界决策''由笔者主导，AI 仅完善代码骨架与报告措辞。

\subsection*{A.3 VTune Flat-AVX2 Avg Frequency 异常的定位}

\paragraph{用户提问。}
VTune 在 Flat-AVX2 上采到 Avg CPU Frequency 只有 1.484 GHz、256-bit FP uOps 只占 26.3\%，这明显不符合 Raptor Lake P-core Turbo 预期。我怀疑是调度把线程迁到了 E-core。请帮我写 PowerShell 命令把被采样进程绑定到 P-core（i9-13900H 的 0--15 号逻辑核对应 P-core），并保持 VTune 的 uarch 采样不受影响。

\paragraph{AI 回复要点。}
(1) 用 \texttt{Start-Process -ProcessorAffinity 0xFFFF} 启动程序，掩码 \texttt{0xFFFF} 覆盖 0--15 号逻辑核；(2) 对 VTune 的 \texttt{amplxe-cl -target-process} 而言，被测进程继承掩码即可，不需要改采样配置；(3) 预期绑核后 Avg Freq 升至 $\sim 5\,\mathrm{GHz}$、Machine Clears 明显下降，因为 E-core 迁移本身会触发 memory disambiguation nuke。

\paragraph{实际采用。}

\subsection*{B.4 真实内核的 objdump 摘要}

\paragraph{Flat-AVX2 内循环（\texttt{flat\_search} @ 0x140001a40）。}
四个独立 256-bit 累加器，每轮处理 32 floats：
\begin{lstlisting}[language={},caption=Flat-AVX2 真实内核]
vmovups ymm4,YMMWORD PTR [rdx]                ; load 8 floats (lane 0)
vmovups ymm5,YMMWORD PTR [rdx+0x20]
sub rdx,-128
vfmadd231ps ymm0,ymm4,YMMWORD PTR [r9-0x80]   ; acc0 += x0*y0
vfmadd231ps ymm2,ymm5,YMMWORD PTR [r9-0x60]   ; acc2 += x1*y1
vmovups ymm4,YMMWORD PTR [rdx-0x40]
vmovups ymm5,YMMWORD PTR [rdx-0x20]
vfmadd231ps ymm1,ymm4,YMMWORD PTR [r9-0x40]
vfmadd231ps ymm3,ymm5,YMMWORD PTR [r9-0x20]
cmp rdx,r10
jne loop_start
\end{lstlisting}
与 4.6 节的 "manual AVX2" 微基准完全一致的风格，验证了内核实际跑的指令流与设计相符。

\paragraph{SQ-AVX2 uint8 L2 内核（\texttt{sq\_search} @ 0x140003a40）。}
256-bit 整数 SIMD：32 uint8/轮，扩展到 16-bit 后用 \texttt{vpmaddwd} 做``平方+相邻对加''：
\begin{lstlisting}[language={},caption=SQ-AVX2 uint8 L2 内核]
vmovdqu    ymm1,[rdx+rax*1]      ; 32 uint8 from x
vmovdqu    ymm2,[rsi+rax*1]      ; 32 uint8 from y
vpunpcklbw ymm0,ymm1,ymm3        ; expand low  16 bytes -> 16 words
vpunpckhbw ymm1,ymm1,ymm3        ; expand high 16 bytes -> 16 words
vpunpcklbw ymm5,ymm2,ymm3
vpunpckhbw ymm2,ymm2,ymm3
vpsubw     ymm0,ymm0,ymm5        ; diff (lo)
vpsubw     ymm1,ymm1,ymm2        ; diff (hi)
vpmaddwd   ymm0,ymm0,ymm0        ; diff^2 + pairwise -> 8 dwords
vpmaddwd   ymm1,ymm1,ymm1
vpaddd     ymm0,ymm0,ymm1
vpaddd     ymm0,ymm0,ymm4        ; running accumulator
\end{lstlisting}
这正是 VTune 中 ``256-bit Int Vector uOps 28.6\%'' 的来源。

\paragraph{PQ-AVX2 LUT 构建内循环（\texttt{pq\_search} @ 0x1400027e0）。}
单累加器 8-float FMA，每轮覆盖 12 维子向量的点积：
\begin{lstlisting}[language={},caption=PQ-AVX2 LUT 构建内核]
vmovups ymm4,YMMWORD PTR [r8+rax*1]
vfmadd231ps ymm0,ymm4,YMMWORD PTR [rcx+rax*1]
add rax,0x20
cmp rax,r10
jne loop_start
\end{lstlisting}
子段维度 $d_{sub}=12$ 使循环计数极小（1--2 轮），单累加器已足够；ADC 查表累加则如正文所述保持标量，这正是 FastScan 的发力点。
