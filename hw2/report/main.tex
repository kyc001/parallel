\section{实验任务}
本次实验围绕 CPU 架构中的两个典型性能问题展开：
\begin{enumerate}
  \item 给定 $n \times n$ 矩阵 $B$ 和向量 $a$，计算每一列与向量的内积
  \[
    s_j = \sum_{i=0}^{n-1} B_{i,j} a_i .
  \]
  重点观察矩阵访问顺序与 cache 局部性的关系。
  \item 给定 $n$ 个浮点数，计算其和
  \[
    S = \sum_{i=0}^{n-1} x_i .
  \]
  重点观察链式依赖、多链独立累加和归约方式对超标量执行的影响。
\end{enumerate}

相较于最初在 WSL 上完成的版本，本次实验迁移到真实 Ubuntu 机器重新执行，代码也进一步整理为多文件工程，而不是把所有逻辑塞进一个源文件中。目录划分如下：
\begin{itemize}
  \item \texttt{src/benchmark.*}：计时、warm-up、绑核与 profiling 封装；
  \item \texttt{src/fs\_compat.hpp}：GCC 7 / GCC 12 的 \texttt{filesystem} 兼容层；
  \item \texttt{src/matrix\_experiment.*}：矩阵列内积的三个实现；
  \item \texttt{src/sum\_experiment.*}：求和的四个实现；
  \item \texttt{src/environment.*}、\texttt{src/profile.*}：环境记录与 profiling 结果导出；
  \item \texttt{src/main.cpp}：实验入口；
  \item \texttt{report/main.tex}：LaTeX 实验报告。
\end{itemize}

\subsection{环境迁移与构建}
从 WSL 迁移到当前机器后，最直接的兼容性问题来自 GCC 版本差异：当前主机是 g++ 7.5.0，标准库中的 \texttt{<filesystem>} 仍需兼容处理。因此本实验加入了 \texttt{src/fs\_compat.hpp} 和 Makefile 中的 \texttt{-lstdc++fs} 逻辑，使工程既能在旧版 GCC 7 上构建，也能继续在新版本 GCC 上运行。

本次实验的构建与执行命令如下：
\begin{lstlisting}[caption={编译、运行与导出汇编}]
make clean all
make run
make asm
\end{lstlisting}

\subsection{测试数据与规模设计}
为便于复现实验与正确性校验，两部分实验都采用固定公式生成数据：
\[
  a_i = 0.5 + ((19i + 13) \bmod 127) / 127.0 ,
\]
\[
  B_{i,j} = 0.2 + ((137i + 23j) \bmod 257) / 257.0 .
\]

矩阵实验选取
\[
  n=\{64,128,256,384,512,768,1024,1536,2048\},
\]
对应工作集大小为 $8n^2$ Byte。这里 $n=256$ 时矩阵大小刚好为 512 KiB，与本机 L2 规模相当；$n=1536$ 时矩阵大小约为 18 MiB，已经超过本机通过 sysfs 读取到的 16 MiB L3 条目。

求和实验选取
\[
  n=\{2^{10},2^{12},2^{14},2^{16},2^{18},2^{20},2^{22},2^{23}\},
\]
数组大小为 $8n$ Byte，用于观察问题规模从 cache 友好区间逐步进入带宽受限区间时，各种实现的性能变化。

\subsection{计时方式}
程序使用 \texttt{std::chrono::steady\_clock} 计时。对每组规模，不预设固定运行次数，而是重复执行待测函数直到累计时间超过 250 ms，再以总时间除以迭代次数得到平均耗时。这样可以明显减小小规模测试中过短运行时间带来的抖动。每次正式计时之前，还会先执行 2 轮 warm-up。

\section{算法设计与分步优化}
\subsection{矩阵列内积}
矩阵部分按“逐步优化”的方式实现了三个版本：
\begin{enumerate}
  \item \textbf{平凡算法}：外层按列循环，内层按行访问 \texttt{matrix[row * n + col]}。该写法与 C/C++ 的行主存储布局不匹配，跨行访问步长为 $n$，空间局部性差。
  \item \textbf{行优先算法}：改为外层按行扫描矩阵，内层按列把该行对所有输出元素的贡献累加到 \texttt{out[col]}。这样内层变成连续访问，更符合 cache line 的利用方式。
  \item \textbf{行优先 + 4 路展开}：在行优先基础上，对内层循环做 4 路手工展开，减少循环控制开销，也让处理器获得更稳定的指令流。
\end{enumerate}

矩阵实验的关键不是减少乘加次数，因为三个版本的计算量同阶，真正拉开差距的是访存模式是否与行主存储布局匹配。循环展开只是第二层优化，收益建立在访问顺序已经被修正的前提之上。

\subsection{求和}
求和部分也按优化阶段实现了四个版本：
\begin{enumerate}
  \item \textbf{链式累加}：单个累加器顺序求和，依赖链最长；
  \item \textbf{两路独立累加}：把一条依赖链拆成两条；
  \item \textbf{四路独立累加}：进一步拆成四条独立链；
  \item \textbf{两两归约}：先把相邻元素配对相加，再不断缩减问题规模直至只剩一个结果。
\end{enumerate}

这里的核心思想是减少串行依赖。链式累加中每一次加法都依赖上一次结果，而两路、四路独立累加把多条累加链分散到不同寄存器上，更容易被超标量流水线并行发射。归约法虽然理论上也缩短了依赖深度，但它需要读写中间结果，因此性能上还要同时接受额外访存成本的检验。

矩阵实验有三点现象最值得注意：
\begin{enumerate}
  \item 当 $n=256$ 时，矩阵大小约为 512 KiB，已经与本机 L2 规模接近。此时行优先和展开版本相较于平凡算法已有 2.8 倍到 3.8 倍提升，说明单纯修正访问顺序就足以产生明显收益。
  \item 当 $n=512$ 以后，矩阵大小达到 2 MiB，已经明显越过 L2。平凡算法时间从 0.201 ms 增长到 1.325 ms，而行优先 + 4 路展开只增长到 0.126 ms，加速比迅速跃升到 10.55 倍，cache 友好访问开始主导差异。
  \item 当 $n=1536$ 和 $n=2048$ 时，工作集进一步逼近并超过本机读取到的 16 MiB L3 条目。此时平凡算法退化尤为明显，而优化版本仍然保持顺滑增长，最终在 $n=2048$ 时取得 31.62 倍的最高加速比。
\end{enumerate}

从结果看，矩阵实验中“循环展开”的收益始终存在，但第一决定因素仍然是访问顺序是否匹配行主存储布局。只有先把访存模式从“跨行跳跃”改成“按行连续扫描”，后续的 4 路展开才真正有发挥空间。

求和实验说明了超标量优化与内存系统约束之间的关系：
\begin{enumerate}
  \item 两路独立累加几乎在所有规模下都稳定维持约 2 倍加速，说明把单条依赖链拆成两条后，处理器基本能稳定并行调度这两组加法。
  \item 四路独立累加表现最好，在 $n=65536$ 时达到本次实验最高加速比 3.88 倍，已经非常接近理想的 4 倍。这说明在数据仍主要受 cache 支持时，多路独立累加能较充分地发挥超标量流水线能力。
  \item 当规模增长到 $2^{22}$ 和 $2^{23}$，数组大小分别为 32 MiB 和 64 MiB，已经明显超过 L3 量级，四路版本的加速比下降到 2.54 倍和 2.73 倍。此时瓶颈开始更多地转移到内存带宽，而不再只是加法依赖链长度。
  \item 两两归约只在很小规模时表现不错，随后很快落后于多链累加；当规模达到 262144 及以上时甚至明显慢于链式累加。原因在于归约虽然减小了依赖深度，但不断读写中间结果，带来了额外访存成本。
\end{enumerate}

此外，不同求和顺序会带来微小浮点舍入差异。实测最大绝对误差出现在 $n=8388608$ 时，量级约为 $4.66\times 10^{-9}$，远小于结果本身，可以视为浮点重排带来的正常现象。

\subsection{汇编观察与 profiling}
对 \texttt{results/assembly\_excerpt.txt} 的观察可以看到，链式累加与多链累加在目标代码层面确实呈现出不同的依赖结构：
\begin{lstlisting}[caption={链式累加与多链累加的关键汇编片段}]
sum_naive:
    vaddsd xmm0, xmm0, QWORD PTR [rdi]

sum_superscalar2:
    vaddsd xmm0, xmm0, QWORD PTR [rax]
    vaddsd xmm1, xmm1, QWORD PTR [rax+0x8]

sum_superscalar4:
    vaddsd xmm1, xmm1, QWORD PTR [rax]
    vaddsd xmm3, xmm3, QWORD PTR [rax+0x8]
    vaddsd xmm0, xmm0, QWORD PTR [rax-0x10]
    vaddsd xmm2, xmm2, QWORD PTR [rax-0x8]
\end{lstlisting}

这与时间结果是吻合的：\texttt{sum\_naive} 只维护一条累加链，而两路、四路版本则显式引入多个彼此独立的累加寄存器，因而更容易被超标量流水线并行发射。

我同时保留了 \texttt{perf\_event\_open} 版本的 profiling 代码，并在 \texttt{results/profile\_results.csv} 中记录结果。但当前环境下所有硬件事件都返回 \texttt{perf\_event\_open failed: No such file or directory}。结合 \texttt{results/environment.txt} 中记录的 \texttt{perf\_event\_paranoid=2} 可以推断，失败原因并不是简单的 paranoid 等级过高，而更像是当前内核/容器环境未向用户态暴露对应的 PMU 事件接口。因而，本报告对性能原因的分析主要依据两类证据：一是不同问题规模下的运行时间趋势，二是反汇编得到的目标代码结构。

\section{结论}
在真实 Ubuntu 机器上重新完成本次实验后，可以得到以下结论：
\begin{enumerate}
  \item 对矩阵列内积问题，决定性能的首要因素是访存模式。将逐列访问改写为行优先访问后，随着工作集越过 L2/L3 边界，加速比持续放大；本次实验在 $n=2048$ 时，行优先 + 4 路展开相对平凡算法达到 31.62 倍加速。
  \item 对求和问题，多链独立累加可以显著缓解串行依赖。两路独立累加稳定接近 2 倍，四路独立累加最高达到 3.88 倍，较好地展示了超标量架构对无依赖指令流的利用能力。
  \item 两两归约并非天然更快。它虽然缩短了依赖深度，但因为引入中间结果写回，在当前实现和当前平台上，当问题规模增大后反而会输给简单的多链累加。
  \item 从 WSL 迁移到真实 Ubuntu 平台后，实验结果更能反映真实硬件行为；同时，工程上也需要考虑 GCC 版本兼容、CPU 可用掩码和 profiling 接口可用性等实际问题。
\end{enumerate}

综合来看，本次实验较为直观地说明了现代 CPU 程序性能并不只由计算量决定。数据访问顺序、指令依赖结构、cache 层次以及系统环境对硬件计数器的支持情况，都会直接影响最终测得的执行时间。
