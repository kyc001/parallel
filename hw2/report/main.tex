\renewcommand{\abstractname}{摘要}
\begin{abstract}
本文在 Windows 11 x86\_64 环境下重新完成 Lab1 CPU 架构编程实验，并使用 Intel VTune Profiler 对关键 workload 做补充剖析。实验围绕矩阵列内积与浮点数组求和两个问题，分别实现了逐列访问、行优先访问、行优先加 4 路展开，以及链式累加、两路独立累加、四路独立累加和两两归约等版本；测试过程中采用固定公式生成数据、单核绑核、warm-up 与累计时长阈值计时，以减小短程序测量抖动。最终结果表明：矩阵实验中，修正访问顺序是决定性优化，在 $n=2048$ 时行优先加 4 路展开相对平凡算法达到 22.41 倍加速；求和实验中，四路独立累加在小规模上最高达到 3.81 倍加速，在 $n=1048576$ 时仍保持 1.93 倍加速，而两两归约在中大规模下明显慢于链式累加。VTune 的 memory-access 与 uarch-exploration 结果进一步说明：矩阵平凡算法具有更强的 DRAM 受限特征，而多链累加能显著改善每周期退休效率，从而把瓶颈从单一依赖链转移到更一般的后端资源约束。

\noindent\textbf{关键词：} CPU 架构编程，cache 局部性，超标量，循环展开，性能测试
\end{abstract}

\section{实验任务}
本次实验围绕 CPU 架构中的两个典型性能问题展开：
\begin{enumerate}
  \item 给定 $n \times n$ 矩阵 $B$ 和向量 $a$，计算每一列与向量的内积
  \[
    s_j = \sum_{i=0}^{n-1} B_{i,j} a_i .
  \]
  重点观察矩阵访问顺序与 cache 局部性的关系。
  \item 给定 $n$ 个浮点数，计算其和
  \[
    S = \sum_{i=0}^{n-1} x_i .
  \]
  重点观察链式依赖、多链独立累加和归约方式对超标量执行的影响。
\end{enumerate}

相较于最初偏向 Linux/\allowbreak VMware 的实现，本次正式提交版本补齐了 Windows 侧的构建、运行与 VTune 采样链路。代码也进一步整理为多文件工程，而不是把所有逻辑塞进一个源文件中。目录划分如下：
\begin{itemize}
  \item \texttt{src/benchmark.*}：计时、warm-up、绑核与 profiling 封装；
  \item \texttt{src/fs\_compat.hpp}：GCC 7 / GCC 12 的 \texttt{filesystem} 兼容层；
  \item \texttt{src/matrix\_experiment.*}：矩阵列内积的三个实现；
  \item \texttt{src/sum\_experiment.*}：求和的四个实现；
  \item \texttt{src/environment.*}、\texttt{src/profile.*}：环境记录与 profiling 结果导出；
  \item \texttt{src/main.cpp}：实验入口；
  \item \texttt{report/main.tex}：LaTeX 实验报告。
\end{itemize}

矩阵实验选取

对应工作集大小为 $8n^2$ Byte。这里 $n=384$ 时矩阵大小约为 1.13 MiB，已经逼近本机 1.25 MiB 的 L2；$n=1536$ 时矩阵大小约为 18 MiB，接近但尚未越过 24 MiB L3；$n=2048$ 时矩阵大小约为 32 MiB，已经超过 L3。

\[
  n=\{2^{10},2^{12},2^{14},2^{16},2^{18},2^{20},2^{22},2^{23}\},
\]
数组大小为 $8n$ Byte，用于观察问题规模从 cache 友好区间逐步进入带宽受限区间时，各种实现的性能变化。

\subsection{计时方式}
程序使用 \texttt{std::chrono::steady\_clock} 计时。对每组规模，不预设固定运行次数，而是重复执行待测函数直到累计时间超过 250 ms，再以总时间除以迭代次数得到平均耗时。这样可以明显减小小规模测试中过短运行时间带来的抖动。每次正式计时之前，还会先执行 2 轮 warm-up。

计时框架的核心思想是“先让一次测量足够长，再计算单次平均开销”，其流程与实际工程代码一致，可概括为：
\begin{lstlisting}[caption={累计时长阈值计时框架},language=C++]
for (int warmup = 0; warmup < 2; ++warmup) {
    do_not_optimize(run_once());
}

size_t iterations = 1;
while (true) {
    auto start = Clock::now();
    double checksum = 0.0;
    for (size_t i = 0; i < iterations; ++i) {
        checksum += run_once();
    }
    auto end = Clock::now();
    double total_ms = duration_ms(end - start);
    if (total_ms >= 250.0) {
        avg_ms = total_ms / iterations;
        break;
    }
    iterations *= 2;
}
\end{lstlisting}

这里将迭代次数按 2 倍递增，可以在小规模问题上快速越过计时精度不够的区间，同时避免人为为每个规模单独手调重复次数。

\section{算法设计与分步优化}
\subsection{矩阵列内积}
矩阵部分按“逐步优化”的方式实现了三个版本：
\begin{enumerate}
  \item \textbf{平凡算法}：外层按列循环，内层按行访问 \texttt{matrix[row * n + col]}。该写法与 C/C++ 的行主存储布局不匹配，跨行访问步长为 $n$，空间局部性差。
  \item \textbf{行优先算法}：改为外层按行扫描矩阵，内层按列把该行对所有输出元素的贡献累加到输出数组中，即 \texttt{out[col]}。这样内层变成连续访问，更符合 cache line 的利用方式。
  \item \textbf{行优先 + 4 路展开}：在行优先基础上，对内层循环做 4 路手工展开，减少循环控制开销，也让处理器获得更稳定的指令流。
\end{enumerate}

矩阵实验的关键不是减少乘加次数，因为三个版本的计算量同阶，真正拉开差距的是访存模式是否与行主存储布局匹配。循环展开只是第二层优化，收益建立在访问顺序已经被修正的前提之上。

对应到代码层面，矩阵实验最关键的变化并不是公式本身，而是循环次序和连续访存方式：
\begin{lstlisting}[caption={矩阵列内积的核心实现片段},language=C++]
// naive: outer loop on columns, inner loop on rows
for (size_t col = 0; col < n; ++col) {
    double acc = 0.0;
    for (size_t row = 0; row < n; ++row) {
        acc += matrix[row * n + col] * vec[row];
    }
    out[col] = acc;
}

// row-major + unroll4: scan a whole row contiguously
for (size_t row = 0; row < n; ++row) {
    const double factor = vec[row];
    const double* row_ptr = matrix + row * n;
    size_t col = 0;
    for (; col + 3 < n; col += 4) {
        out[col]     += row_ptr[col]     * factor;
        out[col + 1] += row_ptr[col + 1] * factor;
        out[col + 2] += row_ptr[col + 2] * factor;
        out[col + 3] += row_ptr[col + 3] * factor;
    }
}
\end{lstlisting}

\subsection{求和}
求和部分也按优化阶段实现了四个版本：
\begin{enumerate}
  \item \textbf{链式累加}：单个累加器顺序求和，依赖链最长；
  \item \textbf{两路独立累加}：把一条依赖链拆成两条；
  \item \textbf{四路独立累加}：进一步拆成四条独立链；
  \item \textbf{两两归约}：先把相邻元素配对相加，再不断缩减问题规模直至只剩一个结果。
\end{enumerate}

这里的核心思想是减少串行依赖。链式累加中每一次加法都依赖上一次结果，而两路、四路独立累加把多条累加链分散到不同寄存器上，更容易被超标量流水线并行发射。归约法虽然理论上也缩短了依赖深度，但它需要读写中间结果，因此性能上还要同时接受额外访存成本的检验。

在实现上，超标量优化最直接的体现就是把一条长依赖链拆成多条独立链：
\begin{lstlisting}[caption={求和实验的关键实现片段},language=C++]
// naive: one dependency chain
double sum = 0.0;
for (size_t i = 0; i < n; ++i) {
    sum += data[i];
}

// superscalar4: four independent accumulators
double s0 = 0.0, s1 = 0.0, s2 = 0.0, s3 = 0.0;
size_t i = 0;
for (; i + 3 < n; i += 4) {
    s0 += data[i];
    s1 += data[i + 1];
    s2 += data[i + 2];
    s3 += data[i + 3];
}
double sum4 = (s0 + s1) + (s2 + s3);
\end{lstlisting}

相较之下，两两归约虽然也缩短了理论依赖深度，但必须不断把中间结果写回数组并重新读取，因此它更依赖具体实现的访存效率。

矩阵实验有三点现象最值得注意：
\begin{enumerate}
  \item 当 $n=384$ 时，矩阵大小已经与本机 L2 规模接近。此时单纯把访问顺序改成行优先，就能带来约 1.77 倍加速；进一步做 4 路展开后，提升到约 2.10 倍。
  \item 当 $n=512$ 以后，矩阵大小达到 2 MiB，已经明显越过 L2。平凡算法时间迅速升高，而行优先与展开版本增长更平缓；到 $n=512$ 时，行优先 + 4 路展开已经达到 9.11 倍加速。
  \item 当 $n=2048$ 时，矩阵大小达到 32 MiB，已经超过本机 24 MiB L3。此时平凡算法平均耗时为 55.48 ms，而行优先 + 4 路展开仅为 2.48 ms，最高加速比达到 22.41 倍。
\end{enumerate}

从结果看，矩阵实验中“循环展开”的收益并非对所有规模都严格单调，但第一决定因素始终是访问顺序是否匹配行主存储布局。只有先把访存模式从“跨行跳跃”改成“按行连续扫描”，后续的 4 路展开才真正有发挥空间。

\begin{enumerate}
  \item 两路独立累加在大多数规模下都稳定接近 2 倍加速，说明把单条依赖链拆成两条后，处理器通常可以较稳定地并行调度这两组加法。
  \item 四路独立累加表现最好，在 $n=4096$ 时达到本次实验最高加速比 3.81 倍；当规模增大到 $n=1048576$ 时仍保持 1.93 倍加速，说明多条独立累加链能够显著缓解串行依赖。
  \item 当规模增长到 $2^{22}$ 和 $2^{23}$，数组大小分别为 32 MiB 和 64 MiB，已经明显超过 L3 量级，四路版本的加速比下降到 1.43 倍和 1.49 倍。此时瓶颈开始更多地转移到 cache miss 与内存带宽，而不再只是加法依赖链长度。
  \item 两两归约只在很小规模时还有一定竞争力，随后很快落后于多链累加；当规模达到 65536 及以上时已经明显慢于链式累加。原因在于归约虽然减小了依赖深度，但不断读写中间结果，带来了额外访存成本。
\end{enumerate}

此外，不同求和顺序会带来微小浮点舍入差异。实测最大绝对误差出现在 $n=8388608$ 时，量级约为 $4.66\times 10^{-9}$，远小于结果本身，可以视为浮点重排带来的正常现象。

\subsection{汇编观察与 profiling}
对 \texttt{results/assembly\_excerpt.txt} 的观察可以看到，链式累加与多链累加在目标代码层面确实呈现出不同的依赖结构：
\begin{lstlisting}[caption={链式累加与多链累加的关键汇编片段}]
sum_naive:
    vaddsd xmm0, xmm0, QWORD PTR [rdi]

sum_superscalar2:
    vaddsd xmm0, xmm0, QWORD PTR [rax]
    vaddsd xmm1, xmm1, QWORD PTR [rax+0x8]

sum_superscalar4:
    vaddsd xmm1, xmm1, QWORD PTR [rax]
    vaddsd xmm3, xmm3, QWORD PTR [rax+0x8]
    vaddsd xmm0, xmm0, QWORD PTR [rax-0x10]
    vaddsd xmm2, xmm2, QWORD PTR [rax-0x8]
\end{lstlisting}

这与时间结果是吻合的：\texttt{sum\_naive} 只维护一条累加链，而两路、四路版本则显式引入多个彼此独立的累加寄存器，因而更容易被超标量流水线并行发射。

\begin{enumerate}

  \item 在求和实验中，\texttt{sum\_naive} 的 CPI 为 0.8308，对应 IPC 约 1.204；\texttt{sum\_superscalar4} 的 CPI 降到 0.5409，对应 IPC 提升到约 1.849。同时 Retiring 比例从 14.5\% 提升到 27.9\%，说明多条独立累加链确实提高了每周期可退休的有效指令数量。
  \item 从热点函数表看，四组 VTune 结果的主要时间都集中在实验核心函数 \texttt{matrix\_dot\_naive}、\texttt{matrix\_dot\_row\_major\_unrolled4}、\texttt{sum\_naive} 和 \texttt{sum\_superscalar4} 上，说明采样结果确实聚焦到了待分析内核，而不是被框架代码稀释。
  \item 需要注意的是，Windows 上 VTune 采样依赖管理员权限和采样驱动，且为保证采样稳定性，我对不同 workload 采用了不同 repeat。因而本节更适合比较“瓶颈类型”和“归一化效率指标”，而不是直接比较不同报告之间的绝对事件总数。
\end{enumerate}

\section{结论}
在 Windows 11 + VTune 环境下重新完成本次实验后，可以得到以下结论：
\begin{enumerate}
  \item 对矩阵列内积问题，决定性能的首要因素仍然是访存模式。将逐列访问改写为行优先访问后，随着工作集越过 L2/L3 边界，加速比持续放大；本次实验在 $n=2048$ 时，行优先 + 4 路展开相对平凡算法达到 22.41 倍加速。
  \item 对求和问题，多链独立累加可以显著缓解串行依赖。两路独立累加稳定接近 2 倍，四路独立累加最高达到 3.81 倍；在 $n=1048576$ 时，四路版本仍有 1.93 倍加速，较好地展示了超标量架构对无依赖指令流的利用能力。
  \item 两两归约并非天然更快。它虽然缩短了依赖深度，但因为引入中间结果写回，在当前实现和当前平台上，当问题规模增大后反而会输给简单的多链累加。
  \item VTune 的结果与计时结论是互相印证的：矩阵平凡算法更受 DRAM 等待约束，多链累加则明显提升了 IPC 和 Retiring 比例。也就是说，本实验不仅在“时间更快”这一层面成立，在“瓶颈为什么变了”这一层面也得到了较直接的 profiling 支持。
  \item 从 Linux/\allowbreak VMware 工作流迁移到当前 Windows 平台后，真正需要解决的并不是算法本身，而是实验工程化问题：平台 API 兼容、管理员权限、采样驱动状态、以及如何让 timing 与 profiling 两条证据链彼此一致。
\end{enumerate}

综合来看，本次实验较为直观地说明了现代 CPU 程序性能并不只由计算量决定。数据访问顺序、指令依赖结构、cache 层次，以及系统环境对 profiling 工具链的支持情况，都会直接影响最终测得的执行时间和我们对瓶颈的解释方式。

这一页主要记录本次复现实验的实际过程，后续可按需要摘入正文。

\begin{enumerate}
  \item 最初工程和交接文档主要基于 Linux/\allowbreak VMware 工作流，报告中仍残留旧的 Ubuntu/AMD 平台描述，因此第一步不是直接跑实验，而是先核对哪些内容已经过时。
  \item 与助教确认后，确定 Windows 平台可以使用 VTune，于是整体策略从“继续补 perf”改成“Windows 重跑主实验 + VTune 补 profiling”，并重新检查课程指导书中关于 VTune 的章节。
  \item 原代码中有大量 Linux 专属实现，例如 \texttt{sched\_setaffinity}、\texttt{perf\_event\_open}、\texttt{/proc} 和 \texttt{/sys} 读取。这一步最大的工作量并不是改算法，而是补平台兼容层，把 Windows 下的绑核、环境记录和外部 VTune 工作流接起来。
  \item Intel 下载页面对命令行抓取并不友好，一开始尝试用脚本直接下载 VTune 遇到了 403；后续确认本机实际上已经安装了 oneAPI VTune 2025.10，只是之前没有从正确路径定位到 \texttt{vtune.exe}。
  \item VTune 采样驱动检查必须以管理员身份执行。通过 \texttt{amplxe-sepreg.exe} 检查后，确认 \texttt{sepdrv5}、\texttt{sepdal}、\texttt{vtss} 服务均已运行，这一步是 Windows 平台上能否采到硬件事件的关键前提。
  \item 批量采集脚本第一次只完成了第一组 workload，后续导出阶段又因为路径处理方式不稳而中断。最后改成“逐项管理员采集 + 普通权限单独导出 CSV”的方式，虽然更慢，但结果可控且容易核对。
  \item 对求和 workload，我尝试过让 \texttt{naive} 和 \texttt{superscalar4} 使用完全相同的 repeat 来做 VTune 对比，但样本太短会导致摘要指标不稳定。因此最终报告采用“较长 steady-state 采样 + 比较 CPI/IPC 和 top-down 比例”的写法，而不强行比较绝对事件总数。
  \item Windows 本机的 timing 会比单纯虚拟机结果更真实，但也更容易受后台任务和驱动活动影响，小规模与中规模实验会出现偶发抖动。因此在 VTune 全部跑完后，我又额外重跑了一轮主实验，选择更平滑的一组数据写入最终报告。
  \item 这次复现里最深的体会是：课程实验不仅考察算法优化，也考察实验方法是否自洽。平台、编译参数、采样权限和报告叙述如果对不上，哪怕代码本身是对的，最终结论也会失真。
\end{enumerate}
