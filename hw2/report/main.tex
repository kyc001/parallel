\section*{摘要}
\addcontentsline{toc}{section}{摘要}
\markboth{摘要}{}

本文围绕两个 CPU 架构实验问题展开：矩阵列内积中的 cache 局部性优化，以及浮点数组求和中的超标量优化。针对矩阵问题，本文实现并比较了逐列访问的平凡算法、行优先的 cache 优化算法以及 4 路循环展开版本；针对求和问题，本文实现并比较了链式累加、两路独立累加、四路独立累加与两两归约四种写法。实验采用固定公式生成测试数据，通过高精度计时与统一累计时长阈值的方法测量平均执行时间，结合工作集规模与 L2/L3 cache 大小分析性能变化。结果表明，行优先算法在大规模下可获得显著加速（$n=2048$ 时行优先达 20.09 倍，加上 4 路展开达 22.41 倍）；四路独立累加最高达 3.81 倍加速。VTune 与汇编分析证实，矩阵优化的主要收益来自降低 Memory Bound，求和优化的主要收益来自缩短依赖链、提高 IPC。

\noindent\textbf{关键词：} cache 局部性，超标量，循环展开，高精度计时，VTune

\section{实验任务}
本次实验包含两个问题：
\begin{enumerate}
  \item 给定 $n \times n$ 矩阵 $B$ 和向量 $a$，计算每一列与向量的内积，比较
  \[
    s_j = \sum_{i=0}^{n-1} B_{i,j} a_i .
  \]
  的两类设计思路：逐列访问的平凡算法与 cache 友好的行优先算法。
  \item 给定 $n$ 个浮点数，计算其和，比较
  \[
    S = \sum_{i=0}^{n-1} x_i .
  \]
  的两类设计思路：链式累加与超标量优化算法。
\end{enumerate}

此外，实验还加入了两项进阶内容：对矩阵算法加入 4 路循环展开，对关键 workload 使用 VTune 与汇编观察分析性能原因。

关闭自动向量化的目的是避免编译器 SIMD 化掩盖 cache 局部性和指令级并行本身的差异；单核绑核用于减少调度抖动。测试时系统电源计划设为"高性能"，未手动关闭 Turbo Boost 和超线程，但通过绑核到单个 P-core 并以累计时长阈值（250\,ms）取平均的方式，尽量削弱频率波动和调度迁移的影响。具体而言，关闭向量化（\texttt{-fno-tree-vectorize}）是禁止跨元素的 SIMD 打包（如 \texttt{vaddpd} 一次处理 4 个 double），但仍允许标量 FMA 指令（如 \texttt{vfmadd213sd}）——这类指令虽融合乘加，但在同一时刻只处理一个数据，属于超标量优化范畴而非向量化。这样做是为了把变量收敛到算法本身的 cache 局部性和指令级并行，避免编译器自动生成的 SIMD 指令把两类优化效果混在一起，使得实验现象更易分析。

\subsection{测试数据与规模设计}
两部分实验均采用固定公式生成数据：
\[
  a_i = 0.5 + ((19i + 13) \bmod 127) / 127.0 ,\quad
  B_{i,j} = 0.2 + ((137i + 23j) \bmod 257) / 257.0 .
\]

矩阵实验选取 $n=\{64,128,256,384,512,768,1024,1536,2048\}$，对应工作集大小为 $8n^2$ Byte。其中 $n=384$ 时矩阵约 1.13\,MiB，逼近 L2；$n=1536$ 时约 18\,MiB，接近 L3；$n=2048$ 时约 32\,MiB，超过 L3。

求和实验选取 $n=\{2^{10},2^{12},2^{14},2^{16},2^{18},2^{20},2^{22},2^{23}\}$，数组大小 $8n$ Byte，用于观察从 cache 友好区间进入带宽受限区间的性能变化。

\subsection{计时方式}
使用 \texttt{std::chrono::steady\_clock} 计时。对每个"算法-规模"组合，先做 2 轮 warm-up，随后从较小重复次数开始运行，若累计时间不足 250\,ms 则重复次数翻倍，直到总测时达到 250\,ms，再以 $t_{\text{avg}} = t_{\text{total}}/\text{repeat}$ 作为单次平均耗时。此外，程序使用 \texttt{asm volatile} 编写的 \texttt{do\_not\_optimize} 和 \texttt{clobber\_memory} 作为反优化屏障，防止编译器消除未使用的计算结果。

\section{算法设计与编程实现}
\subsection{矩阵列内积}
矩阵部分实现了三个版本：（1）\textbf{平凡算法}：外层按列、内层按行，跨步访问局部性差；（2）\textbf{行优先算法}：外层按行、内层按列，连续访存匹配行主布局；（3）\textbf{行优先+4路展开}：在行优先基础上手工展开内层循环。三种算法计算量同阶，性能差异主要来自访存模式。

\begin{lstlisting}[language=C++,caption=平凡算法（逐列访问）]
void matrix_dot_naive(const double* matrix,
    const double* vec, double* out, size_t n) {
  for (size_t col = 0; col < n; ++col) {
    double acc = 0.0;
    for (size_t row = 0; row < n; ++row)
      acc += matrix[row*n + col] * vec[row];
    out[col] = acc;
  }
}
\end{lstlisting}

\begin{lstlisting}[language=C++,caption=行优先算法（cache 友好）]
void matrix_dot_row_major(const double* matrix,
    const double* vec, double* out, size_t n) {
  std::fill(out, out+n, 0.0);
  for (size_t row = 0; row < n; ++row) {
    const double factor = vec[row];
    const double* row_ptr = matrix + row*n;
    for (size_t col = 0; col < n; ++col)
      out[col] += row_ptr[col] * factor;
  }
}
\end{lstlisting}

\begin{lstlisting}[language=C++,caption=行优先 + 4 路展开]
void matrix_dot_row_major_unrolled4(const double* matrix,
    const double* vec, double* out, size_t n) {
  std::fill(out, out+n, 0.0);
  for (size_t row = 0; row < n; ++row) {
    const double f = vec[row];
    const double* rp = matrix + row*n;
    size_t col = 0;
    for (; col + 4 <= n; col += 4) {
      out[col]   += rp[col]   * f;
      out[col+1] += rp[col+1] * f;
      out[col+2] += rp[col+2] * f;
      out[col+3] += rp[col+3] * f;
    }
    for (; col < n; ++col)
      out[col] += rp[col] * f;
  }
}
\end{lstlisting}

\subsection{求和}
求和部分实现了四个版本。核心思想是减少串行依赖：链式累加中每次加法都依赖上一次结果，而多路独立累加把运算分散到多个寄存器上，利于超标量并行发射。

\begin{lstlisting}[language=C++,caption=链式累加]
double sum_naive(const double* data, size_t n) {
  double sum = 0.0;
  for (size_t i = 0; i < n; ++i) sum += data[i];
  return sum;
}
\end{lstlisting}

\begin{lstlisting}[language=C++,caption=四路独立累加]
double sum_superscalar4(const double* data, size_t n) {
  double s0=0, s1=0, s2=0, s3=0;
  size_t i = 0;
  for (; i + 4 <= n; i += 4) {
    s0 += data[i];   s1 += data[i+1];
    s2 += data[i+2]; s3 += data[i+3];
  }
  double sum = (s0+s1) + (s2+s3);
  for (; i < n; ++i) sum += data[i];
  return sum;
}
\end{lstlisting}

\subsubsection{两路独立累加}
将依赖链拆为两条，理论上可将 FP 加法吞吐率提高到接近 2 倍。

\begin{lstlisting}[language=C++,caption=两路独立累加]
double sum_superscalar2(const double* data, size_t n) {
  double s0 = 0.0, s1 = 0.0;
  size_t i = 0;
  for (; i + 2 <= n; i += 2) {
    s0 += data[i];  s1 += data[i+1];
  }
  double sum = s0 + s1;
  for (; i < n; ++i) sum += data[i];
  return sum;
}
\end{lstlisting}

\subsubsection{两两归约}
递归式地将相邻元素配对求和，每轮规模减半，共 $\lceil\log_2 n\rceil$ 轮。当前实现为了保持输入只读，先把数据全量拷贝到 \texttt{scratch}，再在该缓冲区上逐轮覆盖。这样虽然缩短了理论依赖深度，但也引入了一次完整拷贝和后续多轮写回；若允许覆写输入，也可以采用原地 stride 跳跃的树状归约来省去额外缓冲区，不过在单核场景下，这类访存模式通常仍难胜过完全在寄存器内进行的多路独立累加。

\begin{lstlisting}[language=C++,caption=两两归约]
double sum_pairwise(const double* data, size_t n,
                    vector<double>& scratch) {
  copy(data, data+n, scratch.begin());
  size_t active = n;
  while (active > 1) {
    size_t pairs = active / 2;
    for (size_t i = 0; i < pairs; ++i)
      scratch[i] = scratch[2*i] + scratch[2*i+1];
    if (active % 2 == 1)
      scratch[pairs] = scratch[active-1];
    active = pairs + (active % 2);
  }
  return scratch[0];
}
\end{lstlisting}

矩阵实验有三点比较明显：
\begin{enumerate}
  \item 当 $n \leq 384$ 时（工作集 $\leq$ L2），行优先带来约 $1.8 \sim 2.1$ 倍加速，此时 cache miss 差异尚不悬殊。
  \item $n=512$ 以后，矩阵约 2\,MiB 已超过 L2（1.25\,MiB），平凡算法时间急剧升高；到 $n=512$ 时加速比已达 9.11 倍。
  \item $n=2048$ 时矩阵 32\,MiB 超过 L3，平凡算法耗时 55.48\,ms，行优先+展开仅 2.48\,ms，最高 22.41 倍。
\end{enumerate}

第一决定因素始终是访问顺序是否匹配行主布局；循环展开的收益建立在访存已连续化的前提之上。

求和实验说明了超标量优化与内存约束之间的关系：
\begin{enumerate}
  \item 两路独立累加在中小规模下稳定接近 2 倍加速（$n \leq 2^{20}$ 时约 $1.85 \sim 2.07$ 倍），说明拆为两条依赖链后处理器可并行调度；当数据超出 L3 后加速比回落至约 1.4 倍，瓶颈转为内存带宽。
  \item 四路独立累加表现最好，$n=4096$ 时达 3.81 倍；$n=2^{20}$ 时仍保持 1.93 倍。
  \item 当 $n \geq 2^{22}$（$\geq$32\,MiB），超过 L3，四路加速比降至 $\sim$1.5 倍，瓶颈转为内存带宽。
  \item 两两归约在 $n \geq 2^{16}$ 后明显慢于链式累加。当前实现中的 \texttt{scratch} 缓冲区全量拷贝与逐轮写回带来额外访存开销；即便改成原地 stride 归约以减少空间开销，单核下其跨步访存模式也通常难以胜过寄存器内的多路独立累加。
\end{enumerate}

由于浮点数加法不满足严格结合律，中间结果的舍入路径不同，不同求和顺序会产生极小的浮点舍入差异；以链式累加结果为比较基准，最大绝对误差约 $4.66\times 10^{-9}$。

\subsection{汇编观察}
通过 \texttt{objdump -Cd -Mintel} 导出各核心函数的汇编，验证编译器指令流是否符合设计预期。以下摘录关键内循环（已省略 nop 对齐）。

\subsubsection{矩阵：平凡 vs 行优先}

\begin{multicols}{2}
\begin{lstlisting}[language={[x86masm]Assembler},caption=naive 内循环,basicstyle=\ttfamily\tiny,numbers=none]
vmovsd  xmm0, [r10]
vmulsd  xmm0, xmm0, [rax]
add     rax,  0x8
add     r10,  r11     ; +n*8
vaddsd  xmm1, xmm1, xmm0
jne     <loop>
\end{lstlisting}
\columnbreak
\begin{lstlisting}[language={[x86masm]Assembler},caption=row\_major 内循环,basicstyle=\ttfamily\tiny,numbers=none]
vmovsd      xmm0, [r10+rax*8]
vfmadd213sd xmm0, xmm1,
            [rcx+rax*8]
vmovsd      [rcx+rax*8], xmm0
add         rax, 0x1
jne         <loop>
\end{lstlisting}
\end{multicols}

平凡算法 \texttt{add r10, r11} 使指针以 $n \times 8$ 字节跳跃，每次迭代跨越多个 cache line，miss 率显著增高。行优先版本地址连续递增 8 字节，更有利于硬件预取，且编译器自动生成了 FMA 指令（\texttt{vfmadd213sd}），将乘法和加法合为一条。

\subsubsection{矩阵：行优先 + 4 路展开}
\begin{lstlisting}[language={[x86masm]Assembler},caption=unrolled4 内循环,basicstyle=\ttfamily\tiny,numbers=none]
vmovsd      xmm1, [r14]              ; load row[col+0]
vfmadd213sd xmm1, xmm0, [rax]       ; out[col+0] += row[col+0]*f
vmovsd      [rax], xmm1
vmovsd      xmm1, [r14-0x18]         ; load row[col+1]
vfmadd213sd xmm1, xmm0, [rax-0x18]  ; out[col+1] += ...
vmovsd      [rax-0x18], xmm1
vmovsd      xmm1, [r14-0x10]         ; load row[col+2]
vfmadd213sd xmm1, xmm0, [rax-0x10]  ; out[col+2] += ...
vmovsd      [rax-0x10], xmm1
vmovsd      xmm1, [r14-0x8]          ; load row[col+3]
vfmadd213sd xmm1, xmm0, [rax-0x8]   ; out[col+3] += ...
vmovsd      [rax-0x8], xmm1
jne         <loop>
\end{lstlisting}

每次迭代 4 条独立 FMA，步进 32 字节（0x20），每两次迭代恰好消费一整条 cache line（64\,B），循环判断摊销降至 $1/4$。

\subsubsection{求和：链式累加 vs 四路独立累加}

\begin{multicols}{2}
\begin{lstlisting}[language={[x86masm]Assembler},caption=链式累加（单依赖链）,basicstyle=\ttfamily\tiny,numbers=none]
vaddsd xmm0, xmm0, [rcx]
add    rcx, 0x10
vaddsd xmm0, xmm0, [rcx-0x8]
jne    <loop>
\end{lstlisting}
\columnbreak
\begin{lstlisting}[language={[x86masm]Assembler},caption=四路独立累加,basicstyle=\ttfamily\tiny,numbers=none]
vaddsd xmm0, xmm0, [rax]       ; chain 0
vaddsd xmm3, xmm3, [rax+0x8]   ; chain 1
add    rax, 0x20
vaddsd xmm1, xmm1, [rax-0x10]  ; chain 2
vaddsd xmm2, xmm2, [rax-0x8]   ; chain 3
jne    <loop>
\end{lstlisting}
\end{multicols}

链式版本虽被编译器 2 路展开，但两条 \texttt{vaddsd} 均写 \texttt{xmm0}，形成串行依赖链（FP 加法延迟 $\sim$4 周期，IPC $\sim 1/4$）。四路版本使用 \texttt{xmm0}--\texttt{xmm3} 四个独立寄存器，相邻指令无数据依赖，超标量可并行发射，FP add 吞吐理论上可达每周期 1 条。

\subsection{补充：编译器优化级别对比}
在保持 \texttt{-march=native} 和关闭向量化不变的前提下，本文额外对比了 \texttt{-O0} 与 \texttt{-O3} 的性能差异，共 118 组结果，原始数据见仓库 \texttt{results/opt\_levels} 目录。代表性结果（均为 \texttt{-O0} $\to$ \texttt{-O3}）：矩阵 \texttt{naive} 在 $n=2048$ 时由 72.81\,ms 降至 67.96\,ms（1.07 倍），\texttt{row\_major} 由 8.47\,ms 降至 3.45\,ms（2.45 倍）；求和 \texttt{naive} 在 $n=2^{20}$ 时由 2.87\,ms 降至 0.82\,ms（3.48 倍），\texttt{superscalar4} 由 0.90\,ms 降至 0.40\,ms（2.22 倍）。

这些结果说明，编译器优化不能替代算法优化。对大规模 Memory Bound 的矩阵平凡算法，\texttt{-O3} 的收益很有限；而对访存模式已改善或依赖链已拆开的 kernel，\texttt{-O3} 仍可提供约 2--3.5 倍的额外加速。也就是说，当瓶颈仍在循环开销、寄存器分配和指令调度时，编译器优化能显著放大算法设计收益；而当瓶颈已转为主存带宽时，其额外收益就会明显受限。

\section{结论}
本次实验的结果比较一致，可以归纳为以下几点：

\begin{enumerate}
  \item 矩阵问题里，访存顺序是最关键的因素。行优先在 $n=2048$ 时达到 20.09 倍加速，加上 4 路展开后达到 22.41 倍；VTune 中行优先的 CPI 从 3.29 降到 0.34，DRAM Bound 从 79.3\% 降到 17.7\%，LLC miss 也从 29.7M 降到 13.8M（u4）。
  \item 求和问题里，多条独立累加链确实能缓解串行依赖。四路版本最高达到 3.81 倍加速，汇编中可以看到 4 条独立 \texttt{vaddsd} 指令链；VTune 中 CPI 从 0.83 降到 0.54，Core Bound 从 58.2\% 降到 47.7\%。超过 L3 后，瓶颈逐渐转为带宽，加速比也回落到 $\sim$1.5 倍。
  \item 两两归约并不一定更快。当前实现中，\texttt{scratch} 缓冲区带来的全量拷贝与额外读写使 Memory Bound 达到 60.4\%，大规模时反而慢于链式累加；即便改成原地 stride 归约来降低空间复杂度，单核下其不连续访存模式通常仍难胜过寄存器内的多路独立累加。
  \item 从计时结果、汇编和 VTune 指标可以相互印证：矩阵优化的核心是改善 cache 局部性，求和优化的核心是缩短依赖链；循环展开只有在访存已经连续化之后才更容易带来收益。
\end{enumerate}

综合来看，本次实验体现了软硬协同的基本思路：当瓶颈在访存（Memory Bound）时，应优先做 cache 局部性优化，编译器和微架构层面的额外收益有限；当瓶颈转为算力（Core Bound）时，则应通过降低依赖、提高指令级并行度（ILP）进一步释放超标量吞吐。两类优化并非替代关系，而是对应不同层级的性能瓶颈，应结合 profiling 结果有针对性地选择。
