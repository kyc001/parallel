\section{实验内容}
本次实验包含两个部分：
\begin{enumerate}
  \item 计算给定 $n\times n$ 矩阵每一列与给定向量的内积，对比逐列访问的平凡算法与按行访问的 cache 优化算法。
  \item 计算 $n$ 个数的和，对比链式累加、四路独立累加以及两两归约三种实现，分析超标量和访存开销对性能的影响。
\end{enumerate}

本次实验代码位于 \texttt{../src/lab1\_benchmark.cpp}，使用 \texttt{Makefile} 构建，运行后会在 \texttt{../results} 中生成全部原始数据。若课程要求提交 GitHub/Gitee 链接，可在最终提交前将本地代码目录替换为个人仓库地址。

\section{算法设计}
\subsection{矩阵与向量内积}
设矩阵为 $B$，向量为 $a$，输出向量为 $sum$。二维数组在 C/C++ 中按行主序存储，因此不同访问顺序会带来完全不同的空间局部性。

\subsubsection{平凡算法}
逐列访问矩阵。每次固定一列，再沿着行方向跳跃访问：
\begin{lstlisting}[language=C++,caption={矩阵列内积的平凡算法}]
for (size_t col = 0; col < n; ++col) {
    double acc = 0.0;
    for (size_t row = 0; row < n; ++row) {
        acc += matrix[row * n + col] * vec[row];
    }
    out[col] = acc;
}
\end{lstlisting}

该算法的计算量仍然是 $O(n^2)$，但访问步长为 $n$，当矩阵规模超过 cache 容量后，cache miss 会明显增加。

\subsubsection{Cache 优化算法}
改为逐行访问矩阵，使得内层循环连续读取一整行元素：
\begin{lstlisting}[language=C++,caption={矩阵列内积的 cache 优化算法}]
fill(out, out + n, 0.0);
for (size_t row = 0; row < n; ++row) {
    const double factor = vec[row];
    const double* row_ptr = matrix + row * n;
    for (size_t col = 0; col < n; ++col) {
        out[col] += row_ptr[col] * factor;
    }
}
\end{lstlisting}

两种算法的时间复杂度相同，但优化算法的内存访问顺序与行主存储一致，能够更充分利用 cache 的空间局部性。

\subsection{$n$ 个数求和}
\subsubsection{链式累加}
最直接的方法是用一个累加器顺序求和：
\begin{lstlisting}[language=C++,caption={链式累加}]
double sum = 0.0;
for (size_t i = 0; i < n; ++i) {
    sum += data[i];
}
\end{lstlisting}

该算法结构最简单，但每次加法都依赖前一次结果，容易形成长依赖链。

\subsubsection{四路独立累加}
把一个累加器拆为四个独立累加器，再在末尾合并：
\begin{lstlisting}[language=C++,caption={四路独立累加}]
double s0 = 0.0, s1 = 0.0, s2 = 0.0, s3 = 0.0;
size_t limit = n - (n % 4);
for (size_t i = 0; i < limit; i += 4) {
    s0 += data[i];
    s1 += data[i + 1];
    s2 += data[i + 2];
    s3 += data[i + 3];
}
double sum = (s0 + s1) + (s2 + s3);
\end{lstlisting}

这样做等价于循环展开和多链累加的结合。单条依赖链被拆成四条更短的依赖链，更容易被超标量流水线并行调度。

\subsubsection{两两归约}
按树形结构两两相加，每轮把问题规模缩小一半：
\begin{lstlisting}[language=C++,caption={两两归约}]
copy(data, data + n, scratch.begin());
size_t active = n;
while (active > 1) {
    size_t pairs = active / 2;
    for (size_t i = 0; i < pairs; ++i) {
        scratch[i] = scratch[2 * i] + scratch[2 * i + 1];
    }
    active = pairs;
}
\end{lstlisting}

这种方法同样减少了长依赖链，但会额外读写辅助数组，访存开销较大，因此未必在实际机器上更快。

\section{实验方案}
\subsection{测试数据}
两部分实验都采用固定公式生成输入，便于重复验证：
\begin{itemize}
  \item 向量元素：\texttt{0.25 + ((17i + 11) mod 97) / 97.0}
  \item 矩阵元素：\texttt{0.1 + ((131i + 17j) mod 251) / 251.0}
\end{itemize}

\subsection{规模设计}
\begin{itemize}
  \item 矩阵实验取 $n=\{64,128,256,384,512,768,1024,1536,2048\}$。
  \item 求和实验取 $n=\{2^{10},2^{12},2^{14},2^{16},2^{18},2^{20},2^{22},2^{23}\}$。
\end{itemize}

矩阵所占空间约为 $n^2\times 8$ Byte。以本机 L2 为 1.25 MiB、L3 为 24 MiB 为参照，$n=384$ 时矩阵约 1.125 MiB，已经接近单核 L2；$n=1536$ 时矩阵约 18 MiB，已经逼近 L3 范围。这些规模点适合观察 cache 层次变化带来的转折。

\subsection{计时方法}
采用 \texttt{std::chrono::steady\_clock} 进行高精度计时。对于每组规模，不固定执行次数，而是重复执行核心函数直到总计时超过 250 ms，再计算单次平均时间。这样可以减弱短程序带来的计时误差。

此外，程序在测试开始前会先进行两轮 warm-up，并将测试线程绑定到逻辑核 0，以减少调度带来的波动。

可以看到：
\begin{itemize}
  \item 当 $n\le 384$ 时，两种算法差距不算大，加速比约为 $1.4\times$ 到 $2.8\times$。此时矩阵规模还没有明显超出单核 L2，平凡算法虽然按列访问，但 cache 仍能部分容纳热点数据。
  \item 当 $n=512$ 时，矩阵大小约为 2 MiB，已经明显超过单核 L2，加速比突然提升到 $7.38\times$。
  \item 在 $n=1024$ 时，优化算法达到本次实验最高加速比 $10.69\times$。此时平凡算法的跨行步长访问导致大量无效 cache line 装载，而逐行访问能够持续命中连续内存。
  \item 当 $n=1536$ 和 $n=2048$ 时，矩阵规模已经接近甚至超过共享 L3，加速比略有回落，但仍稳定在 $8\times$ 以上，说明访问顺序的影响仍然非常显著。
\end{itemize}

该实验的核心结论是：在计算量相同的情况下，访存模式决定了 cache 是否能真正发挥作用。逐列访问与内存布局相悖，导致大量 cache miss；逐行访问则把空间局部性利用到了最大。

对求和实验可以得到三点结论：
\begin{itemize}
  \item 四路独立累加在小规模数据上效果明显，最高加速比出现在 $n=4096$，达到 $3.67\times$。原因是原始链式算法只有一条长依赖链，而四路累加把依赖分散到四个寄存器中，更容易被超标量流水线并行执行。
  \item 随着数据规模增大到数百万级，加速比逐渐下降到约 $1.1\times$ 到 $2.0\times$。此时瓶颈从“算术依赖”逐渐转向“内存带宽”，因此纯粹的指令级并行优化收益被削弱。
  \item 两两归约在本机上始终不如链式累加，尤其在大规模数据时明显更慢。根本原因是该算法需要反复写回辅助数组，虽然减少了依赖链，但引入了更高的访存成本。
\end{itemize}

此外，四路累加和两两归约会改变浮点数的求和顺序，因而与链式累加出现了极小的数值差异。实测误差量级约为 $10^{-13}\sim10^{-10}$，属于浮点运算舍入误差的正常范围。

\subsection{汇编层面的简单验证}
使用 \texttt{objdump -Cd -Mintel} 查看可执行文件，可以直接看到链式累加和四路累加在汇编层面的结构差异。关键片段如下：

\begin{lstlisting}[caption={求和函数的关键汇编片段}]
sum_naive:
    vaddsd xmm0, xmm0, QWORD PTR [rdi]
    add    rdi, 0x8
    cmp    rax, rdi
    jne    ...

sum_superscalar4:
    vaddsd xmm0, xmm0, QWORD PTR [rax]
    vaddsd xmm3, xmm3, QWORD PTR [rax+0x8]
    vaddsd xmm1, xmm1, QWORD PTR [rax-0x10]
    vaddsd xmm2, xmm2, QWORD PTR [rax-0x8]
\end{lstlisting}

可以看到，链式累加只有一个累加寄存器 \texttt{xmm0}，每条加法依赖上一条指令的结果；而四路累加同时使用 \texttt{xmm0/xmm1/xmm2/xmm3} 四个寄存器维护独立的累加链，处理器可以更容易发射多条彼此无依赖的标量加法指令。这与前述时间结果是一致的。

\section{结论}
本次实验完成了两类 CPU 架构相关优化的实现与测试，得到如下结论：
\begin{enumerate}
  \item 对矩阵列内积问题，改变访问顺序本身就可以带来非常显著的性能收益。随着问题规模跨越 L2/L3 cache 边界，按行访问相对逐列访问的优势会迅速扩大，本机最高达到约 $10.69\times$。
  \item 对 $n$ 个数求和问题，超标量友好的多链累加能显著缩短依赖链，在小规模数据上最高达到约 $3.67\times$ 加速。但当数据规模继续增大时，瓶颈转向内存带宽，加速比会明显收敛。
  \item 两两归约虽然减少了依赖深度，但在当前实现中需要频繁写辅助数组，访存开销抵消了理论优势，因此整体不如四路独立累加。
\end{enumerate}

综合来看，本次实验说明了一个很重要的事实：现代 CPU 上程序性能不只由计算量决定，内存访问模式、指令依赖关系以及编译器生成代码的结构同样决定最终表现。
