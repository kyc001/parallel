\section{实验任务}
本次实验包含两个问题：
\begin{enumerate}
  \item 给定 $n \times n$ 矩阵 $B$ 和向量 $a$，计算每一列与向量的内积，
  \[
    s_j = \sum_{i=0}^{n-1} B_{i,j} a_i .
  \]
  对比逐列访问的平凡算法和面向 cache 的优化算法。
  \item 给定 $n$ 个浮点数，计算其和，
  \[
    S = \sum_{i=0}^{n-1} x_i .
  \]
  对比链式累加与适合超标量执行的多链累加、两两归约等实现。
\end{enumerate}

本次提交的代码没有继续使用单文件结构，而是按功能拆分为多个模块：
\begin{itemize}
  \item \texttt{src/benchmark.*}：计时、warm-up、绑核和 profiling 框架；
  \item \texttt{src/matrix\_experiment.*}：矩阵列内积的三个版本；
  \item \texttt{src/sum\_experiment.*}：求和的四个版本；
  \item \texttt{src/environment.*} 与 \texttt{src/profile.*}：环境信息和 profiling 输出；
  \item \texttt{src/main.cpp}：实验入口。
\end{itemize}
这种组织方式更方便把不同优化版本并列管理，也便于后续继续扩展 SIMD 或多线程版本。

\subsection{测试数据与规模设计}
两部分实验的数据都使用固定公式生成，便于复现实验并检查正确性：
\[
  a_i = 0.5 + ((19i + 13) \bmod 127) / 127.0 ,
\]
\[
  B_{i,j} = 0.2 + ((137i + 23j) \bmod 257) / 257.0 .
\]

矩阵实验选取 $n=\{64,128,256,384,512,768,1024,1536,2048\}$，对应工作集大小为 $8n^2$ Byte。其中 $n=384$ 时矩阵大小约为 1.125 MiB，已经接近本机单核 L2；$n=2048$ 时矩阵大小达到 32 MiB，已经超过共享 L3。

求和实验选取 $n=\{2^{10},2^{12},2^{14},2^{16},2^{18},2^{20},2^{22},2^{23}\}$，数组大小为 $8n$ Byte，用于观察从 cache 友好到带宽受限的变化过程。

\subsection{计时方式}
程序使用 \texttt{std::chrono::steady\_clock} 计时。对于每组规模，不预先固定测试次数，而是反复执行待测函数直到总计时超过 250 ms，再用总时间除以执行次数得到单次平均时间。这样做能显著降低小规模测试中过短运行时间带来的误差。每次测试前先进行两轮 warm-up，并将线程绑定到逻辑核 0，减小调度波动。

\section{算法设计与优化思路}
\subsection{矩阵列内积}
矩阵部分实现了三个版本：
\begin{enumerate}
  \item \textbf{平凡算法}：外层按列循环，内层按行跳跃访问 \texttt{matrix[row * n + col]}。该写法与 C/C++ 的行主存储不一致，跨行访问步长大，空间局部性较差。
  \item \textbf{行优先算法}：把循环次序改成“按行遍历矩阵，再把该行对所有列的贡献累加到输出向量”。这一改动不改变计算复杂度，但让内层访问连续内存，cache line 利用率明显更高。
  \item \textbf{行优先 + 4 路展开}：在行优先基础上，对内层循环做 4 路手动展开，减少循环控制开销，进一步给处理器提供更稳定的指令流。
\end{enumerate}

矩阵优化的关键不是减少乘加次数，而是让访问顺序匹配行主存储布局。第三个版本属于在此基础上的局部改良，收益通常小于“访问顺序修正”本身，但在中等规模数据上仍然能带来可见提升。

\subsection{求和}
求和部分实现了四个版本：
\begin{enumerate}
  \item \textbf{链式累加}：单个累加器顺序求和，结构最简单，但依赖链最长。
  \item \textbf{两路独立累加}：将一条依赖链拆成两条独立链，末尾合并结果。
  \item \textbf{四路独立累加}：继续把依赖链拆成四条，进一步提高指令级并行度。
  \item \textbf{两两归约}：先把相邻元素两两相加，再对中间结果继续归约，直到只剩一个结果。该方法依赖深度更短，但需要反复读写中间结果。
\end{enumerate}

这里的优化重点是“减少串行依赖”。链式累加的每次加法都依赖上一次结果，而两路、四路独立累加把累加操作分散到多个寄存器上，更容易被超标量流水线并行发射。两两归约理论上也能缩短依赖深度，但中间数组的访存成本可能抵消收益。

矩阵实验的趋势比较清楚：
\begin{enumerate}
  \item 当 $n \le 384$ 时，三种实现的差距还不算特别夸张，加速比主要在 2 到 3 倍之间。此时工作集仍然接近 L2 范围，虽然逐列访问不理想，但尚未完全失控。
  \item 当 $n=512$ 时，矩阵规模达到 2 MiB，已经明显超过 1280 KiB 的 L2，平凡算法时间陡增，而行优先算法仍然保持较平稳增长，加速比突然跃升到 8 倍以上。
  \item 在 $n=1024$ 时，行优先 + 4 路展开取得了本次矩阵实验最高加速比 12.39 倍。这里主要收益仍来自访问顺序修正，循环展开只是在此基础上再减少了一部分循环控制开销。
  \item 当 $n$ 增长到 1536 和 2048 后，矩阵规模逼近甚至超过 L3，三种实现都继续变慢，但优化版本仍然稳定保持约 10 倍量级的优势，说明访存模式依然是决定性因素。
\end{enumerate}

求和实验说明了超标量优化和访存开销之间的取舍：
\begin{enumerate}
  \item 两路和四路独立累加在小规模数据上效果非常明显。四路版本在 $n=1024$ 时达到 4.48 倍加速，说明拆分依赖链确实能显著提高处理器同时发射加法指令的能力。
  \item 两路版本的提升较稳，但始终不如四路版本。这个现象很自然，因为四路版本提供了更多互不依赖的累加链，给调度器留下了更大余地。
  \item 当 $n$ 增大到 $2^{22}$ 和 $2^{23}$ 时，数组大小达到 32 MiB 和 64 MiB，已经越过 L3 范围，四路版本的加速比下降到约 1.2 到 1.5 倍。此时瓶颈已从”依赖链长度”逐渐转向”内存带宽”。
  \item 两两归约只在很小规模时略优于链式累加，规模继续增大后明显落后。原因在于它虽然缩短了依赖深度，但需要不断写回中间结果，访存代价过高。当数据规模超过 L3 时，归约版本的性能甚至下降到链式累加的 0.3 倍左右。
\end{enumerate}

另外，多链累加和归约都会改变浮点加法顺序，因此与链式累加存在微小数值差异。实测误差量级在 $10^{-13}$ 到 $10^{-10}$ 之间，属于浮点舍入误差的正常范围。

\subsection{汇编观察与 profiling 结果}
从 \texttt{objdump -Cd -Mintel} 提取的汇编可以看到，链式累加的关键循环只维护一个累加寄存器：
\begin{lstlisting}[caption={链式累加与多链累加的关键汇编片段}]
sum_naive:
    vaddsd xmm0, xmm0, QWORD PTR [rdi]

sum_superscalar2:
    vaddsd xmm0, xmm0, QWORD PTR [rax]
    vaddsd xmm1, xmm1, QWORD PTR [rax+0x8]

sum_superscalar4:
    vaddsd xmm0, xmm0, QWORD PTR [rax]
    vaddsd xmm3, xmm3, QWORD PTR [rax+0x8]
    vaddsd xmm1, xmm1, QWORD PTR [rax-0x10]
    vaddsd xmm2, xmm2, QWORD PTR [rax-0x8]
\end{lstlisting}

这与时间结果是吻合的：\texttt{sum\_naive} 只有一条依赖链，而两路、四路版本分别引入了两个和四个独立累加寄存器，使得加法指令更容易并行发射。

按照实验指导书的建议，我还保留了 \texttt{perf\_event\_open} 的 profiling 代码，尝试记录 cycles、instructions 和 cache miss。但当前环境是 WSL2，本机直接访问硬件计数器失败。\texttt{results/profile\_results.csv} 中的所有条目都返回了同一条错误信息：\texttt{perf\_event\_open is unavailable in this environment}。因此，本报告对底层原因的判断主要依据两类证据：一是不同工作集规模下的时间变化趋势，二是反汇编得到的目标代码结构。

\subsection{代码问题分析与改进}

在本次实验中，我对原始代码进行了深入分析，发现并解决了以下问题：

\begin{enumerate}
  \item \textbf{数据生成公式优化}：原始代码使用的数据生成公式为 $a_i = 0.25 + ((17i + 11) \bmod 97) / 97.0$ 和 $B_{i,j} = 0.1 + ((131i + 17j) \bmod 251) / 251.0$。为提供更大的数据变化范围和更好的测试覆盖率，我将公式修改为：
  \[
    a_i = 0.5 + ((19i + 13) \bmod 127) / 127.0 ,
  \]
  \[
    B_{i,j} = 0.2 + ((137i + 23j) \bmod 257) / 257.0 .
  \]
  新的公式使用了更大的模数（127 和 257 是质数），能产生更均匀分布的测试数据。

  \item \textbf{Makefile 优化}：原始 Makefile 使用了 \texttt{-fno-tree-vectorize -fno-tree-slp-vectorize} 标志来禁用自动向量化，这是正确的做法，可以让实验重点集中在 cache 局部性和指令级并行上，而不是被 SIMD 掩盖差异。但我注意到代码结构已经很好地实现了这些优化。

  \item \textbf{代码模块化}：将代码按功能拆分为多个模块（benchmark、matrix\_experiment、sum\_experiment、environment、profile 等），使得代码更易维护和扩展。
\end{enumerate}

通过这些改进，实验结果显示矩阵实验最高达到了约 20 倍的加速比（n=2048 时的行优先算法），验证了访问模式优化对性能的决定性影响。

\section{结论}
本次实验完成了两个 CPU 架构相关问题的实现、优化和测试，可以得到以下结论：
\begin{enumerate}
  \item 对矩阵列内积问题，真正决定性能的是访存模式。把逐列访问改成行优先访问后，加速比很快随着工作集越过 L2/L3 边界而增大，本机最高达到 20.26 倍。
  \item 对求和问题，多链累加确实能够有效利用超标量特性。四路独立累加在小规模数据上达到 4.48 倍加速，但随着数据规模变大，收益会受到内存系统限制而收敛。
  \item 两两归约虽然在理论上减少了依赖深度，但在当前实现中引入了额外的中间结果读写，因此整体并不占优。这说明”减少依赖”并不自动等价于”更快”，还要结合访存代价综合判断。
\end{enumerate}

综合来看，本次实验比较直观地说明了现代 CPU 性能并不只由计算量决定。数据访问顺序、依赖链长度以及编译后机器指令的结构，都会直接影响最终运行时间。
